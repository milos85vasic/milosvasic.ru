フック

自分でメールサーバを運用することは、システム管理の中でも最も過酷な仕事の一つとして悪名高い。SMTP、IMAP、POP3、TLS、DKIM、SPF、ファイアウォール、証明書、SELinux、そして互いに協調し合わなければメールが届かない十数ものデーモンたち──。Mail Server Factoryの売り文句は実に率直だ。「メールサーバをボスのように運用せよ」。ユーザーはシンプルなJSONファイルで望む構成を記述するだけで、Factoryがそれを解釈し、ターゲットマシン上に強固でDocker化されたスタックを一から構築してくれる。

なぜ魅力的なのか

このプロジェクトの魅力は、メールサーバのデプロイメントをコンパイル問題として扱う点にある。JSON設定ファイルがソースコード、Factoryがコンパイラ、そしてターゲットOS上で稼働する疎結合のDockerベースメールスタックが出力だ。各コンポーネントが疎結合かつコンテナ化されているため、後からスケーリングしやすいクリーンな基盤が生まれる。手作業で編集された設定ファイルの迷路とは対照的だ。

さらに「便利」から「本格的」へと押し上げるのが、その適用範囲の広さだ。Factoryは12種類の接続方式──SSH、Docker、Kubernetes、AWS SSM、Azure Serial Console、GCP OS Login、Libvirt、カスタムプロトコル、データベース、ファイルシステム、クラウドプロバイダ、コンテナランタイム──を通じてスタックをデプロイできる。対応OSは25種類のLinuxディストリビューションに及び、多くのツールが無視するものも含まれる。西側の主要ディストリ（Ubuntu、Debian、CentOS、Fedora、AlmaLinux、Rocky、openSUSE）はもちろん、ロシア系（ALT、Astra、ROSA）、中国系（openEuler、openKylin、Deepin）も網羅している。この幅広さは明確なメッセージだ──「あなたのメールサーバ、あなたのハードウェア、あなたの管轄権」。

難題

第一の難題は、「同じインストール」が決して同じではないことだ。パッケージ名、initシステム、ファイアウォールのフロントエンド、SELinuxの挙動は、25種類のディストリビューションごとに異なる。Factoryには本格的なセキュリティフレームワークが組み込まれている──CertificateValidator、DockerCredentialsManager、SELinuxChecker、PasswordValidator、ConnectionPoolなどだ。これらの機能はプラットフォームごとに動作が異なるため、どれも前提として扱うことはできない。

第二の難題は接続層そのものだ。SSH経由でターゲットに到達するのと、AWS SSM、Azure Serial Console、Libvirt経由で到達するのでは、まるで違う。各トランスポートには独自の認証方式、レイテンシ、障害モードがある。これら12種類すべてを一つのインストールパイプラインに抽象化し、インストール手順が「どのようにバイトがボックスに届くか」を気にしないようにする──これがプロジェクトのアーキテクチャの中核だ。

第三の難題は、最初にクリーンなマシンを用意することだ。Factoryは、preseed、kickstart、cloud-init、autoyastを使って、対応するすべてのディストリビューションで無人OSインストールを自動化する。scripts/qemu_manager.shを通じてQEMU仮想マシンを制御し、ISOのチェックサム検証も行う。さらに企業向けの工夫として、双方向SMBキャッシュを備えている。これは、ネットワーク共有から不足しているイメージを取得し、逆に不足しているものをアップロードし、最後の手段としてインターネットからダウンロードする仕組みだ。

ゲームチェンジャーとなる理由

このプロジェクトが革新的なのは、自前でホストするメールサーバを再現可能かつ監査可能にした点だ。メールインフラはまさに、データとして記述され、同じように再構築できるものであってほしい。深夜2時に記憶を頼りに組み立てるものではない。スタックをJSONで定義し、セキュリティ体制をコードで強制する──AES-256-GCM暗号化、12文字以上のSSH鍵パスフレーズの強制、ポート25/587/465/993/995向けの自動ファイアウォールルール、証明書検証とHSTSを備えたTLS、監査ログ、RBAC──これにより、「本番レベルのメールサーバ」は数週間のプロジェクトから、レビュー・バージョン管理・再実行が可能な設定ファイルへと変わる。

また、このプロジェクトは「理想」ではなく「実証済み」だ。439件のテストが100%合格、SonarQubeの品質ゲートも100%クリア、さらにディストリビューションをまたいだSMTP/IMAP/POP3の実際の動作検証やスループット・レイテンシ測定を含む包括的なテストフレームワークを備えている。メールを預けるに足るツールであるためには、このような証拠こそが最大の機能なのだ。

最も難しかった部分の解決方法

私は「何を構築するか」「どうやってターゲットに到達するか」「どのOSを使うか」を分離した。JSON設定ファイルは意図を記述し、トランスポート抽象化レイヤーが12種類の接続方式を一つのインターフェースに隠蔽する。これにより、インストール手順はSSH経由であろうとAWS SSM経由であろうと、同じ論理操作を発行できる。また、ディストリビューションごとのロジックは一箇所に集約され、25種類のディストリマトリックスが各所に漏れ出すことはない。このレイヤリングのおかげで、新しいディストリビューションや接続方式を追加しても、コードベース全体に影響が波及しない。

セキュリティに関しては、運用者の善意に任せることを拒否した。フレームワークが厳格な部分を強制する──SELinuxのチェック、証明書の検証、Docker認証情報の管理、脆弱なパスワードやパスフレーズなしSSH鍵の拒否──これにより、正しいデプロイメントがデフォルトとなり、慎重な作業の成果ではなくなる。シェルは重いプロビジョニングに適したツールとして使い、オーケストレーションとセキュリティフレームワークにはKotlin/JVMを採用。各コンポーネントはDockerによって疎結合を保っている。

そして、ほとんどのインストーラが後回しにする「クリーンなマシン」のブートストラップ問題も解決した。対応するすべてのディストリビューションで無人インストールを自動化し、テストターゲット用にQEMU仮想マシンを自動起動。さらに双方向SMB ISOキャッシュにより、同じイメージを25回も再ダウンロードすることなく、フリートデプロイメントを実現する。すべてのISOは、どこから取得したものであろうとチェックサム検証が行われる。その結果、裸のメタル（あるいは裸のVM）から、強固でテスト済みの稼働中メールサーバまで、たった一つのファイルから実現できるツールが完成した。

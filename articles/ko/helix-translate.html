후크

어떤 형식의 책이든 HelixTranslate에 건네고 목표 언어를 지정하기만 하면 됩니다. HelixTranslate는 여러분의 텍스트를 가장 강력한 검증된 LLM으로 전달하고, 실시간 대시보드에 진행 상황을 스트리밍하며, 품질을 몰래 저하시키지 않습니다. REST API, HTTP/3 지원, 실시간 WebSocket 이벤트를 갖춘 고성능 엔터프라이즈급 범용 전자책 번역 도구입니다.

매력적인 이유

번역 도구는 보통 특정 제공자를 선택하게 하고 오늘만큼은 괜찮기를 바랄 뿐입니다. HelixTranslate는 이 방식을 뒤집었습니다. 기본 경로는 브리지 선택 방식입니다. LLMsVerifier 프로젝트와 통합되어 가장 강력한 검증된 API 모델과 순위가 매겨진 대체 체인을 자동으로 확보합니다. 사용자가 제공자를 선택할 필요가 없습니다. 시스템은 현재 검증된 최적의 모델을 선택하고, 최상위 선택지가 실패하면 순위 목록을 따라 대체 모델로 전환합니다.

이 핵심 기능 주변에는 실시간 모니터링 시스템이 구축되어 있습니다. WebSocket 허브는 웹 대시보드에 실시간 진행 상황, 이벤트, 오류를 스트리밍하며, 고유 ID를 가진 여러 동시 번역 세션을 지원하고, 다중 클라이언트 연결 시 자동 재연결을 처리합니다. 터미널에 막혀 있는 화면을 응시하는 대신, 긴 책이 문단 단위로 번역되는 과정을 지켜볼 수 있습니다. OpenAI, Anthropic, DeepSeek, Zhipu, Qwen, Gemini 등 다양한 제공자에 접근할 수 있으며, 모두 검증된 경로로 연결됩니다.

어려운 문제들

첫 번째 어려운 문제는 실패 시 정직성입니다. API 키가 없거나 제공자가 실패했을 때 쉬운 방법은 약한 로컬 모델로 조용히 대체하고 모든 것이 정상인 척하는 것입니다. 그러면 성공한 것처럼 보이는 쓰레기 번역이 나옵니다. 저는 정반대의 원칙을 엄격히 적용했습니다. 제공자 API 키가 설정되지 않으면 브리지는 정직한 오류를 반환하며, 절대 조용히 로컬 대체 모델로 전환하지 않습니다. 이를 전체 파이프라인에 일관되게 적용하는 것은 단순한 한 줄짜리 체크가 아닌 설계상의 제약입니다.

두 번째 문제는 SSH를 통한 분산 번역입니다. 큰 책은 순차적으로 번역하면 느리기 때문에, 시스템은 원격 SSH 워커를 지원합니다. 안전하게 연결하고, 번역 작업을 배포하며, 원격 측에서 진행 상황을 추적하고, 오류와 대체 모델 처리를 담당합니다. 원격 워커를 조정하면서도 대시보드에 일관된 실시간 이벤트를 전송하는 것은 수백 가지 작은 문제로 깨지기 쉬운 통합 작업입니다.

세 번째 문제는 실시간 이벤트 처리입니다. 여러 동시 세션의 실시간 진행 상황을 여러 대시보드 클라이언트에 스트리밍하는 것은 단순한 진행률 표시가 아닙니다. 재연결, 세션 기록, 세션별 추적 기능을 갖춘 진정한 WebSocket 아키텍처 문제입니다.

게임 체인저인 이유

검증된 최적 모델 전략이 바로 훔쳐갈 만한 아이디어입니다. 정적인 제공자 선택을 신뢰하는 대신, HelixTranslate는 외부 검증을 지속적으로 활용해 실제로 성능이 뛰어난 모델로 라우팅하며, 성능이 떨어지면 결정론적인 대체 체인으로 전환합니다. 이는 “어떤 LLM을 번역에 사용해야 할까?”라는 질문을 추측에서 측정 가능한 자동 결정으로 바꿔줍니다. 또한 이 도구는 코드 변경 없이도 기본 모델 환경이 변화함에 따라 더 나아집니다.

또한 의도적으로 범위를 제한했습니다. 한때 프로젝트는 로컬 런타임 제공자(Ollama, LlamaCpp)와 SSH 로컬 번역 경로를 지원했지만, 이후 브리지 단계에서 이들을 모두 제거했습니다. 이제 이들을 선택하면 조용히 저하된 로컬 모델을 실행하는 대신 “더 이상 지원되지 않음”이라는 정직한 오류를 반환합니다. 이는 로드맵에도 동일한 정직성 원칙을 적용한 것입니다. 저는 조용히 나쁜 결과를 내는 것보다 기능을 제거하는 것을 택했습니다.

가장 어려운 부분을 해결한 방법

아키텍처의 핵심은 번역과 LLMsVerifier를 연결하는 브리지 패키지(pkg/bridge)입니다. 제공자 선호도를 하드코딩하는 대신, 번역기는 브리지에 가장 강력한 검증된 모델과 순위가 매겨진 대체 체인을 요청한 후 이를 기반으로 번역합니다. 이것이 “제공자를 선택할 필요가 없다”는 말이 실제로 구현된 방식이며, 여기서 조용한 대체 모델 전환을 금지하는 규칙이 적용됩니다. 브리지는 로컬 모델로 대체하지 않고 명시적인 오류를 반환합니다.

실시간 모니터링을 위해 전담 패키지로 역할을 분리했습니다. 연결 및 브로드캐스트 관리를 담당하는 websocket 허브, 이벤트 스트림을 정의하고 전송하는 events 시스템, 원격 실행을 처리하는 sshworker 패키지입니다. 독립 실행형 모니터 서버는 WebSocket 스트림을 소비해 대시보드를 구동하므로, 번역 실행과 모니터링이 분리됩니다. CLI는 이벤트를 전송하고, 서버는 이를 팬아웃하며, 여러 대시보드 클라이언트가 자동 재연결로 구독합니다.

분산 처리를 위해 SSH 워커 계층은 안전한 연결 관리와 원격 실행을 처리하면서 진행 상황을 동일한 이벤트 시스템을 통해 보고합니다. 따라서 대시보드 입장에서는 원격 번역이 로컬 번역과 동일하게 보입니다. 전체 시스템은 HTTP/3를 지원하는 REST API로 노출되며, 포괄적인 WebSocket 모니터링 테스트 스위트로 뒷받침됩니다. 관찰할 수 없는 번역 시스템은 신뢰할 수 없는 번역 시스템이기 때문입니다.

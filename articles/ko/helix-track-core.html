후크

JIRA와 Confluence를 처음부터 하나로 통합해, Go 마이크로서비스로 재구축하고, 암호화된 데이터베이스 위에서 초당 수만 건의 요청을 처리할 만큼 빠른 성능을 구현한다고 상상해 보세요. 이것이 바로 HelixTrack Core입니다. “자유를 위한 오픈소스 JIRA 대안”이라는 슬로건을 내건 이 프로젝트는 단순한 모방품이 아닙니다. 에픽, 서브태스크, 스프린트, 작업 로그, 커스텀 필드, 대시보드, 권한 체계, 그리고 Confluence 스타일의 문서 엔진까지—모든 기능을 단일한 REST API로 제공합니다.

매력적인 이유

대부분의 프로젝트 관리 도구는 시간이 지나면서 수십 개의 엔드포인트가 난립하는 거대한 모놀리스로 변모합니다. 저는 정반대의 길을 선택했습니다. HelixTrack Core는 액션 기반 라우팅을 적용한 통합 /do 엔드포인트를 제공합니다. 클라이언트는 액션 이름과 페이로드를 전송하면 서버가 일관된 미들웨어와 핸들러 파이프라인을 통해 이를 처리합니다. 이 단 하나의 설계 결정이 수백 가지 기능—이슈, 애자일 보드, 투표, 알림 체계, 활동 스트림, 댓글 멘션 등—에 걸쳐 API의 일관성을 유지하면서도 대규모 API에서 흔히 발생하는 엔드포인트 난립 문제를 해결합니다.

그 위에 구축된 것은 진정한 Confluence 대안입니다. Documents V2 확장 기능은 스페이스, HTML/마크다운/일반 텍스트로 작성된 리치 페이지, 전체 버전 히스토리(차이점 비교 및 롤백 지원), 인라인 댓글, @멘션, 반응, 템플릿, 분석 기능을 제공합니다. 이 모든 것은 자체 데이터베이스 테이블로 지원되며 수백 개의 모델 테스트로 검증되었습니다. 하나의 시스템을 구축하는 것도 프로젝트지만, 두 시스템을 하나의 바이너리 안에서 협력하는 모듈로 구현하는 것이야말로 흥미로운 부분입니다.

어려운 문제들

이 프로젝트를 진정으로 어렵게 만드는 세 가지 핵심 과제가 있습니다.

첫째, 성능 저하 없는 보안입니다. 기업용 이슈 트래커는 민감한 데이터를 다루기 때문에 데이터베이스는 SQLCipher(AES-256)를 통해 암호화되어 있습니다. 암호화는 보통 지연 시간을 유발하지만, 이 프로젝트의 엔지니어링 도전 과제는 높은 동시성 환경에서도 쿼리 응답 속도를 유지하면서 암호화 오버헤드를 최소화하는 것이었습니다.

둘째, 다중 데이터베이스 현실입니다. 개발자는 로컬 환경에서 SQLite를 선호하지만, 운영 환경에서는 동시성과 내구성이 뛰어난 PostgreSQL을 사용합니다. 두 엔진을 모두 지원하기 위해 PLpgSQL 수준의 최적화와 버전별 마이그레이션(V1부터 V4까지, 90개 이상의 테이블로 확장된 스키마)을 적용해야 했습니다. 이는 모든 기능이 두 가지 매우 다른 엔진에서 정확하게 작동해야 한다는 의미입니다.

셋째, 정확성을 유지한 실시간 협업입니다. 여러 사람이 동시에 동일한 티켓을 편집할 때 발생하는 분산 상태 문제—업데이트 손실과 충돌—를 해결해야 합니다. 사용자가 서로의 작업에 방해받지 않으면서도 이러한 문제를 해결하려면 신중한 동시성 모델이 필수적입니다.

게임 체인저인 이유

HelixTrack Core는 권한을 교체 가능한 일급 엔진으로 취급합니다. 계층적 권한 상속을 지원하는 컨텍스트 기반 계층적 권한을 구현했으며, 권한 시스템은 플러그인 방식으로 동작합니다. 즉, 서비스의 내부에서 실행할 수도 있고, 외부 HTTP 인증 서비스로 위임할 수도 있습니다. 이러한 분리는 조직이 기존의 ID 및 정책 인프라에 트래커를 통합할 수 있도록 해주며, 내장된 모델과 싸울 필요가 없게 합니다.

또한 대부분의 트래커가 간과하는 자동 서비스 탐색 기능을 제공합니다. 클라이언트는 UDP 브로드캐스트를 통해 로컬 네트워크에서 Core 서버를 자동으로 발견하므로, 웹, 데스크톱, 안드로이드, iOS 등 다양한 플랫폼의 클라이언트가 수동 설정 없이 연결할 수 있습니다. JWT 인증, 시간 추적, 문서, 채팅을 위한 확장 시스템, 커밋을 티켓에 매핑하는 Git 통합 기능까지 더해지면, 이는 단순한 앱이 아닌 플랫폼처럼 동작합니다.

가장 어려운 부분을 해결한 방법

API의 경우, 저는 초기에 액션 라우팅 패턴을 확정했습니다. 모든 작업은 동일한 /do 라이프사이클을 거칩니다—인증 및 권한 미들웨어를 통과한 후, 액션 이름으로 키가 지정된 핸들러가 처리합니다. 이것이 바로 시스템이 새로운 기능을 추가해도(1단계: 우선순위 및 해결 상태, 2단계: 에픽 및 작업 로그, 3단계: 투표 및 알림 체계, 4단계: 병렬 편집) API가 파편화되지 않는 이유입니다. 새로운 기능은 새로운 액션으로 등록되며, 전송 계층과 미들웨어는 변하지 않습니다.

동시 편집의 경우, 낙관적 잠금과 버전 충돌을 구현하고, 완전한 변경 히스토리 레이어를 추가했습니다. 편집 가능한 모든 엔티티는 버전을 가지며, 오래된 버전을 대상으로 한 쓰기 작업은 거부되고 클라이언트가 해결할 수 있는 충돌로 처리됩니다. V4 스키마에서 추가된 히스토리 테이블은 모든 변경 내역을 기록합니다. 여기에 명시적 엔티티 잠금 관리와 충돌 해결 액션을 추가하고, 기존 WebSocket 레이어와 연동하여 실시간 업데이트를 제공합니다. 이를 통해 사용자는 다른 사람의 편집을 덮어쓰지 않으면서도 실시간 협업이 가능합니다.

암호화와 속도 사이의 긴장 관계에 대해서는, 저장소 계층에서 SQLCipher를 통한 투명한 AES-256 암호화를 활용하면서 핫 패스를 캐싱으로 처리하여 일반적인 읽기 패턴이 암호화된 저장소에서도 빠르게 동작하도록 설계했습니다. 이 코드베이스는 수천 개의 자동화된 테스트(각 단계별로)로 뒷받침되며, 90개 이상의 테이블 스키마와 암호화된 다중 데이터베이스, 동시 편집 시스템에 변경 사항을 적용하더라도 동작의 신뢰성을 유지할 수 있게 합니다. 아키텍처 다이어그램, 사용자 매뉴얼, 배포 가이드까지 코드와 함께 제공됩니다. 운영할 수 없는 JIRA 대안은 진정한 대안이 아니기 때문입니다.

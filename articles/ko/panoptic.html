후크

UI 자동화는 파편화되어 있습니다. 웹에는 Selenium이나 Playwright를, 모바일에는 Appium과 XCUITest를, 데스크톱에는 또 다른 도구를 배워야 하며, 화면 녹화를 원한다면 별도의 레코더까지 필요합니다. 플랫폼마다 새로운 도구, 새로운 추상화, 새로운 불안정성의 원인이 존재합니다. ’파놉틱’(Panoptic, ’모든 것을 보는 눈’이라는 뜻)은 이러한 복잡한 생태계를 하나의 Go 프레임워크로 통합해 웹, 데스크톱, 모바일 전반에 걸쳐 자동화, 스크린샷, 세션 녹화를 단일한 프로그래밍 모델로 수행할 수 있게 합니다.

매력적인 이유

파놉틱이 주목받는 이유는 ’소프트웨어를 제어하는 것’과 더불어 ’소프트웨어의 동작을 관찰하는 것’을 동등한 핵심 기능으로 다루기 때문입니다. 단순히 버튼을 클릭하는 데 그치지 않고, 고품질의 타임스탬프가 찍힌 스크린샷과 다양한 형식의 전체 세션 영상을 녹화합니다. 즉, 하나의 테스트 실행으로 검증 결과와 시각적 증거를 동시에 얻을 수 있습니다. 녹화 기능은 부가적인 요소가 아니라 프레임워크의 일부로 설계되었습니다.

진정한 크로스 플랫폼을 지원합니다. 웹 자동화는 Chrome, Firefox, Safari, Edge를 아우르며, 모바일은 iOS와 Android 앱 자동화와 화면 녹화를 지원합니다. 데스크톱에서는 UI 자동화와 화면 캡처가 가능합니다.这一切는 깔끔한 Go API로 제공됩니다—web.NewBrowser(), browser.Navigate(...), browser.Screenshot(...), 사용자 정의 CSS 및 XPath 선택자, 명시적 대기 전략 등—뿐만 아니라 CLI로 일반적인 사용 사례를 처리할 수 있습니다(panoptic record --platform ios --device iPhone13 --output mobile_demo.mp4). 동일한 개념의 API가 헤드리스 Chrome 뷰포트부터 녹화된 iPhone 세션까지 연결된다는 점이 핵심입니다.

어려운 문제들

첫 번째 어려운 문제는 ’요소’라는 개념이 플랫폼마다 다르다는 점입니다. 웹의 버튼은 CSS나 XPath로 접근 가능한 DOM 노드지만, iOS의 컨트롤은 접근성 트리에 존재하며, 데스크톱 위젯은 OS 네이티브 핸들입니다. 파놉틱은 네 가지 완전히 다른 ’이 요소를 찾아 탭하라’는 방언을 구사하면서도 통합된 요소 탐지 및 상호작용 모델을 제공해야 합니다.

두 번째는 타이밍 문제입니다. UI는 어디에서나 비동기적이지만, 그 방식은 각기 다릅니다. 파놉틱은 명시적 대기 전략을 제공합니다—타임아웃이 설정된 WaitForElement와 임의의 조건을 폴링하는 WaitForCondition—왜냐하면 하드 슬립(hard sleep)은 자동화 불안정성의 주범이기 때문입니다. div.loading이 사라지기를 기다리는 것과 네이티브 뷰가 렌더링되기를 기다리는 것이 동일한 방식으로 동작하도록 대기 프레임워크를 설계하는 것은 섬세한 작업입니다.

세 번째는 자동화를 수행하면서 녹화를 병행하는 문제입니다. 고품질의 30fps 영상을 설정 가능한 뷰포트로 캡처하되, 녹화 중인 자동화의 타이밍에 영향을 주지 않아야 합니다. 이는 특히 모바일에서 어려운 과제입니다. 모바일에서는 자신이 소유한 캔버스가 아니라 기기나 시뮬레이터 화면을 녹화해야 하기 때문입니다.

게임 체인저인 이유

진정한 변화는 통합과 증거 제공에 있습니다. 모든 플랫폼의 자동화가 하나의 프레임워크에서 이루어지면, 팀은 네 가지 대신 하나의 멘탈 모델만 학습하면 되고, 시각적 증거를 무료로 얻을 수 있습니다. 불안정한 테스트는 더 이상 들여다봐야 하는 스택 트레이스가 아니라, 직접 시청할 수 있는 영상이 됩니다. 이것만으로도 팀이 실패를 분석하는 방식이 달라집니다.

또한 CI/CD 파이프라인에 쉽게 통합될 수 있도록 설계되었습니다. 확장 가능한 플러그인 아키텍처, 설정 및 해제를 위한 before/after 훅, 브라우저와 녹화, 플랫폼별 모바일 설정을 위한 YAML 구성 파일, 그리고 명시적인 CI/CD 통합 기능까지 제공합니다. 로컬에서 실행한 것과 동일한 테스트 스위트를 CI에 연결할 수 있으며, 실패한 빌드에 첨부되는 아티팩트(스크린샷과 MP4 파일)는 정확히 원하는 형태입니다.

가장 어려운 부분을 해결한 방법

의도적으로 Go를 선택했습니다. 크로스 플랫폼 자동화는 내부적으로 많은 동시 I/O 작업—브라우저 드라이버, 기기 브리지(Android의 ADB, iOS의 Xcode 도구), 녹화 인코더와의 통신—을 동시에 처리해야 합니다. Go의 고루틴과 채널은 ’이 고루틴에서 UI를 제어하고, 저 고루틴에서 화면 프레임을 캡처하며, 또 다른 고루틴에서 대기 조건을 모니터링한다’는 복잡한 작업을 효율적으로 조율할 수 있게 해주며, 단일 정적 바이너리로 CLI를 CI에 배포하는 것도 간단합니다.

네 가지 요소 모델을 통합하기 위해 플랫폼 드라이버들이 공통의 상호작용 인터페이스를 구현하도록 했고, 선택자는 최소 공통 분모가 아닌 풍부한 표현력을 유지하도록 설계했습니다. 웹에서는 CSS와 XPath를, 모바일과 데스크톱에서는 네이티브 로케이터를 사용해 추상화로 인해 필요한 정밀한 선택자를 포기하지 않도록 했습니다. 대기 프레임워크는 이 인터페이스 위에 구축되어, WaitForCondition이 어떤 플랫폼에서든 동일한 방식으로 동작합니다.

녹화 중 왜곡을 방지하기 위해 캡처를 자동화의 핵심 경로 외부에서 독립적으로 실행되는 별도의 파이프라인으로 취급했습니다(panoptic.yaml에서 형식, 품질, fps 설정). 또한 훅과 플러그인 시스템을 처음부터 도입해 확장성을 확보했습니다. 자동화에서 확실한 한 가지는 ’내가 예상하지 못한 무언가가 필요해질 것’이라는 점이며, 프레임워크는 이를 강제 포크 없이 유연하게 수용할 수 있어야 합니다.

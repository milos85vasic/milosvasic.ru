Крючок

Представьте, что вы перестраиваете JIRA и Confluence с нуля как один микросервис на Go, и делаете его достаточно быстрым, чтобы обрабатывать десятки тысяч запросов в секунду с зашированной базой данных внизу. Это и есть HelixTrack Core: “Открытая альтернатива JIRA для свободного мира”. Это не игрушечная копия — она поставляется с эпиками, подзадачами, спринтами, журналами работы, пользовательскими полями, панелями управления, схемами разрешений и полным движком документов в стиле Confluence, все за одной единой REST API.

Почему это fascinates

Большинство инструментов отслеживания проектов растут в разветвленные монолиты с десятками конечных точек, которые расходятся за годы. Я пошел другим путем. HelixTrack Core暴ляет единую конечную точку /do с маршрутизацией на основе действий: клиенты отправляют именованное действие и полезную нагрузку, и сервер диспетчеризирует его через последовательную цепочку middleware и обработчиков. Это одно решение по проектированию сохраняет поверхность API согласованной на сотнях операций — вопросы, доски Agile, голосование, схемы уведомлений, потоки активности, упоминания в комментариях — без разрастания конечных точек, которое делает большие API не поддерживаемыми.

На вершине этого находится настоящая альтернатива Confluence. Расширение Documents V2 реализует пространства, богатые страницы в HTML/Markdown/простом тексте, полную историю версий с разностями и откатом, комментарии в строке, упоминания, реакции, шаблоны и аналитику — все это поддерживается своим собственным набором таблиц базы данных и проверяется сотнями модульных тестов. Построение одной из этих систем — это проект; построение обоих как кооперирующих модулей в одном бинарном файле — это интересная часть.

Трудные проблемы

Три вещи делают это действительно трудным.

Во-первых, безопасность без налог на производительность. Корпоративные отслеживатели вопросов содержат конфиденциальные данные, поэтому база данных зашифрована на месте с помощью SQLCipher (AES-256). Шифрование обычно означает штраф за задержку — инженерная задача заключалась в том, чтобы сохранить этот overhead минимальным, одновременно обслуживая запросы быстро под тяжелой конкурентностью.

Во-вторых, реальность нескольких баз данных. Разработчики хотят SQLite для бесшовной локальной настройки; производство хочет PostgreSQL с его конкурентностью и долговечностью. Поддержка обоих — включая оптимизации на уровне PLpgSQL и схему, которая эволюционировала через несколько версий миграций (V1 через V4, выросла за девяносто таблиц) — означает, что каждая функция должна быть правильной на двух очень разных движках.

В-третьих, взаимодействие в реальном времени с правильностью. В момент, когда несколько человек редактируют один и тот же тикет, вы сталкиваетесь с классической проблемой распределенного состояния: потерянные обновления и конфликтующие записи. Решение этой проблемы без блокировки пользователей из работы друг друга требует намеренной модели конкурентности, а не после мысли.

Что делает его игроком

HelixTrack Core рассматривает разрешения как первый класс, заменимый движок. Он реализует иерархические разрешения на основе контекста с наследованием, и реализация разрешений является подключаемой — вы можете запустить ее локально внутри службы или делегировать внешней службе HTTP-авторизации. Это разделение позволяет организациям вставить отслеживатель в существующую инфраструктуру идентификации и политики вместо борьбы с встроенной моделью.

Он также делает что-то, что большинство отслеживателей игнорируют: автоматическое обнаружение службы. Клиенты находят серверы Core в локальной сети через UDP-broadcast, поэтому много платформенные клиенты (веб, настольные, Android, iOS) могут подключиться без ручной настройки. В сочетании с аутентификацией JWT, системой расширений для отслеживания времени, документов и чата, и интеграцией Git, которая сопоставляет коммиты с тикетами, он ведет себя как платформа, а не как отдельное приложение.

Как я решил самые трудные части

Для API я привержен шаблону маршрутизации на основе действий рано. Каждая операция проходит через одну и ту же жизнь /do — middleware аутентификации и разрешений, затем обработчик, ключевой по имени действия. Это почему система может продолжать добавлять возможности (приоритеты и решения Phase 1, эпические и журналы работы Phase 2, голосование и схемы уведомлений Phase 3, параллельное редактирование Phase 4) без фрагментации API. Новые функции регистрируют новые действия; транспорт и middleware остаются постоянными.

Для редактирования в реальном времени я реализовал оптимистическое блокирование с конфликтами версий плюс полный слой истории изменений. Каждая редактируемая сущность несет версию; запись, нацеленная на устаревшую версию, отклоняется с конфликтом, который клиент может решить, и таблицы истории (добавленные в схему V4) записывают полный след изменений. Я добавил явное управление блокировкой сущностей и действия разрешения конфликтов поверх, и подключил все это к существующему слою WebSocket, чтобы сотрудники видели обновления в реальном времени. Это дает взаимодействие в реальном времени без молчаливого переписывания чьего-то редактирования.

Для напряжения между шифрованием и скоростью дизайн опирается на SQLCipher для прозрачного AES-256 на уровне хранилища, одновременно проталкивая горячие пути через кэширование, так что общие шаблоны чтения остаются быстрыми даже с зашированной базой данных внизу. Кодовая база поддерживается большим набором автоматических тестов — более тысячи тестов на фазы — что позволяет мне вносить изменения в схему более чем из девяноста таблиц и зашифрованной, много базы данных, редактируемой одновременно системы и все еще доверять, что поведение сохраняется. Диаграммы архитектуры, руководство пользователя и руководство по развертыванию поставляются вместе с кодом, потому что альтернатива JIRA, которую никто не может эксплуатировать, не является альтернативой вовсе.

钩子

试想一下，从零开始重建 JIRA 和 Confluence，将它们整合为一个 Go 微服务，并使其足够快速，能在底层加密数据库的支持下每秒处理数万次请求。这就是 HelixTrack Core ——「自由世界的开源 JIRA 替代方案」。它不是一个玩具般的克隆产品，而是真正提供了史诗级任务、子任务、冲刺、工作日志、自定义字段、仪表盘、权限方案，以及一个完整的 Confluence 风格文档引擎，所有功能均通过单一 REST API 对外开放。

为何令人着迷

大多数项目跟踪工具会随着时间演变成庞大的单体应用，拥有数十个逐渐分化的接口。而我反其道而行。HelixTrack Core 仅暴露一个统一的 /do 端点，采用基于动作的路由：客户端发送一个命名动作及其负载，服务器通过一致的中间件和处理程序管道进行分发。这一设计决策使得 API 在数百种操作（问题、敏捷看板、投票、通知方案、活动流、评论提及等）中保持一致性，避免了大型 API 因接口膨胀而难以维护的问题。

在此基础上，还构建了一个真正的 Confluence 替代方案。Documents V2 扩展实现了空间、HTML/Markdown/纯文本的富文本页面、完整的版本历史（包括差异对比和回滚）、内联评论、@提及、反应、模板及分析功能——所有这些均由独立的数据库表支撑，并通过数百个模型测试验证。构建其中一个系统已是一项工程，而将两者作为协同模块整合进同一个二进制文件，才是真正有趣的部分。

真正的难题

有三个问题让这项工作变得异常艰巨。

首先，安全性与性能的平衡。企业级问题跟踪器存储着敏感数据，因此数据库采用 SQLCipher（AES-256）进行静态加密。加密通常意味着性能损耗——工程挑战在于如何在高并发下保持查询速度，同时将开销降至最低。

其次，多数据库的现实。开发者希望使用 SQLite 实现无缝本地部署，而生产环境则需要具备高并发和持久性的 PostgreSQL。同时支持两者——包括 PLpgSQL 级别的优化和一个历经多版本迁移（从 V1 到 V4，已扩展至九十多张表）的架构——意味着每个功能都必须在两种截然不同的引擎上保持正确性。

第三，实时协作中的正确性。当多个用户同时编辑同一工单时，便会面临经典的分布式状态问题：更新丢失和写入冲突。要在不阻塞用户相互操作的情况下解决这一问题，需要一个精心设计的并发模型，而非事后补救。

颠覆性的意义

HelixTrack Core 将权限视为一等公民，且支持可替换的引擎。它实现了基于上下文的层级权限，支持继承，且权限实现可插拔——既可在服务内部本地运行，也可委托给外部 HTTP 授权服务。这种分离使得组织能够将跟踪器无缝接入现有的身份和策略基础设施，而无需与内置模型硬碰硬。

此外，它还做到了大多数跟踪器忽视的一点：自动服务发现。客户端通过 UDP 广播在局域网中自动发现 Core 服务器，因此多平台客户端（Web、桌面、Android、iOS）无需手动配置即可连接。结合 JWT 认证、时间跟踪/文档/聊天的扩展系统，以及将提交映射到工单的 Git 集成，它更像一个平台而非单一应用。

最难部分的解决之道

在 API 设计上，我早早确立了动作路由模式。每个操作都经过相同的 /do 生命周期——认证与权限中间件，然后由动作名称索引的处理程序。这正是系统能够不断添加功能（第一阶段的优先级和解决方案、第二阶段的史诗和工作日志、第三阶段的投票和通知方案、第四阶段的并行编辑），而 API 却不至于碎片化的原因。新功能注册新动作，传输和中间件保持不变。

对于并发编辑，我实现了带版本冲突的乐观锁，并构建了完整的变更历史层。每个可编辑实体都携带版本号；针对过期版本的写入会被拒绝，并返回客户端可解决的冲突，而历史表（在 V4 架构中添加）则记录了完整的变更轨迹。在此基础上，我进一步增加了显式的实体锁管理和冲突解决动作，并将整个机制与现有的 WebSocket 层集成，使协作者能实时看到更新。这确保了实时协作不会悄无声息地覆盖他人的编辑。

在加密与速度的矛盾中，设计倾向于依赖 SQLCipher 实现透明的存储层 AES-256 加密，同时通过缓存优化热点路径，使常见的读取模式即使在加密存储下也能保持快速。代码库由庞大的自动化测试套件支撑——跨阶段的测试超过千项——这正是我能够在拥有九十多张表、加密、多数据库、并发编辑的复杂系统中进行修改，并依然对其行为保持信心的原因。架构图、用户手册和部署指南与代码一同发布，因为一个无人能操作的 JIRA 替代方案，终究算不上替代方案。

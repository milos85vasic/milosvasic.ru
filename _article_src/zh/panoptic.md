## 钩子

UI 自动化支离破碎。你学 Selenium 或 Playwright 来操控网页，Appium 和 XCUITest 来驱动移动端，桌面端又是另一套工具，如果想录制操作过程，还得再找个单独的录屏器。每跨越一个平台，就多一个新工具、新抽象、新的不稳定因素。Panoptic——以「全视之眼」命名——将这整个动物园般的工具链压缩为一个 Go 框架，用同一套编程模型实现网页、桌面和移动端的自动化操作、截图与录屏。

## 为何引人入胜

Panoptic 的魅力在于，它将「观察软件行为」视为与「驱动软件」同等重要的核心能力。它不仅仅是点击按钮，还能捕捉高质量、带时间戳的截图，并以多种格式录制完整的会话视频。这意味着一次测试运行既能生成断言结果，又能提供直观的视觉证据——录屏不是附加功能，而是框架的有机组成部分。

它真正做到了跨平台通吃。网页自动化覆盖 Chrome、Firefox、Safari 和 Edge；移动端支持 iOS 和 Android 应用自动化，并带录屏功能；桌面端则提供 UI 自动化和屏幕捕捉。所有功能均通过简洁的 Go API 暴露：`web.NewBrowser()`、`browser.Navigate(...)`、`browser.Screenshot(...)`，支持自定义 CSS 和 XPath 选择器，以及明确的等待策略。此外，还提供 CLI 工具处理常见场景（如 `panoptic record --platform ios --device iPhone13 --output mobile_demo.mp4`）。同一套概念 API 能从无头 Chrome 视口延伸至 iPhone 录屏会话，这正是其核心价值所在。

## 难点所在

第一个难点在于「元素」在每个平台上的定义截然不同。网页按钮是可通过 CSS 或 XPath 定位的 DOM 节点；iOS 控件存在于辅助功能树中；桌面组件则是操作系统原生句柄。Panoptic 需在表层提供统一的元素检测与交互模型，底层却要与四种完全不同的「找到并点击」方言打交道。

第二个难点是时序。UI 在所有平台上都是异步的，但异步的方式各不相同。Panoptic 内置了明确的等待策略——带超时的 `WaitForElement`，以及轮询任意条件直至满足的 `WaitForCondition`——因为硬性睡眠是自动化不稳定的头号元凶。设计出无论是等待 `div.loading` 消失还是原生视图渲染完成，行为都完全一致的等待原语，是一项精细的工作。

第三个难点是边操作边录制。以高质量捕捉 30fps 视频，同时支持可配置的视口，且不干扰正在录制的自动化操作时序，是一项真正的性能与同步挑战——尤其是在移动端，录制的是设备或模拟器屏幕，而非自有画布。

## 颠覆性所在

颠覆性在于「整合+证据」。当自动化在一个框架内覆盖所有平台时，团队只需构建一套思维模型，而非四套，且能免费获得视觉证据。一个不稳定的测试不再是让人眯眼琢磨的堆栈跟踪，而是一段可直接观看的视频。仅此一点，就彻底改变了团队排查故障的方式。

它还专为流水线集成而设计：可扩展的插件架构、用于设置与清理的前后钩子、用于配置浏览器、录制和移动平台设置的 YAML 文件，以及明确的 CI/CD 集成接口。你可以将同一套测试套件无缝接入 CI，与本地运行一致，而生成的工件——截图和 MP4 文件——正是你希望附加到失败构建中的内容。

## 如何攻克最棘手的部分

我刻意选择了 Go。跨平台自动化在底层涉及大量并发 I/O——同时与浏览器驱动、设备桥接（Android 的 ADB、iOS 的 Xcode 工具）和录制编码器通信。Go 的 goroutine 和 channel 让「在一个 goroutine 上驱动 UI，在另一个上捕获屏幕帧，在第三个上监控等待条件」的协调变得可行且高效，而单一静态二进制文件则让 CLI 工具轻松部署到 CI 环境。

为了统一四种元素模型，我让平台驱动实现了共同的交互接口，并保持选择器的表达力而非最小公分母——网页端支持 CSS 和 XPath，移动端和桌面端则使用原生定位器——确保抽象层不会牺牲你实际需要的精确选择器。等待原语正是基于这一接口构建，因此无论条件检查的是哪个平台，`WaitForCondition` 的行为都完全一致。

为了实现无失真的录制，我将捕获流程设计为独立配置的管道（在 `panoptic.yaml` 中设置格式、质量、fps），使录制器与自动化操作并行运行，而非嵌入其关键路径。此外，我从一开始就通过钩子和插件系统构建了整个框架的可扩展性，因为自动化领域唯一确定的一点是：总有人需要实现我未曾预见的功能——而框架应当允许他们这样做，而非强迫他们 fork 代码。

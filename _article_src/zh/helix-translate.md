## 吸睛之处

无论以何种格式交付书稿，只要指定目标语言，HelixTranslate 便能全文翻译——将文本自动分发至当前最强的 *已验证* 大语言模型，实时推送进度至监控面板，且绝不在背后悄然降低质量。这是一款高性能、企业级的通用电子书翻译工具包，提供 REST API、HTTP/3 支持及实时 WebSocket 事件通知。

## 为何引人入胜

翻译工具通常要求你自行选择供应商，并祈祷它当下足够优秀。HelixTranslate 反其道而行。其默认路径采用 **桥接选择机制**：通过与我的 LLMsVerifier 项目集成，自动调用最强的已验证 API 模型，并按评分顺序构建备选链路。你无需选择供应商，系统会自动挑选当前已验证的最佳模型，若首选模型失效，则按优先级顺序回退。

围绕这一核心，系统还配备了完整的 **实时监控体系**：WebSocket 中心实时推送翻译进度、事件及错误至网页面板，支持多个并发翻译任务（各自拥有唯一 ID），并处理多客户端连接及自动重连。你能实时查看长篇书稿逐段翻译的过程，而非盯着卡死的终端界面。它对接多家供应商——OpenAI、Anthropic、DeepSeek、智谱、通义千问、Gemini 等——所有模型均通过验证后可用。

## 难点所在

第一个难点是 **失效时的诚实**。当 API 密钥缺失或供应商失效时，最简单的做法是悄然切换至弱小的本地模型，假装一切正常。这会产生看似成功实则劣质的翻译。我反其道而行，将其设为硬性规则：若未配置供应商 API 密钥，桥接模块将直接返回明确的错误——绝不存在静默回退至本地模型的情况。在整个流程中严格执行这一规则，是一项设计约束，而非一行代码的检查。

第二个难点是 **基于 SSH 的分布式翻译**。大部头书籍串行翻译耗时过长，因此系统支持远程 SSH 工作节点——安全连接、分发翻译任务、跟踪远端进度，并处理错误与回退。在保持实时事件流向面板一致性的同时，协调远程工作节点，这类集成往往会在无数细节上崩溃。

第三个难点是 **实时事件流的正确构建**。为多个并发任务向多个面板客户端实时推送进度——支持重连、会话历史及每个任务的独立跟踪——这绝非简单的进度条问题，而是真正的 WebSocket 架构挑战。

## 颠覆性所在

"已验证最佳模型"策略是值得借鉴的核心理念。HelixTranslate 不依赖静态供应商选择，而是通过持续的外部验证，将任务路由至当前 *实际表现最佳* 的模型，并在其失效时按预设链路回退。这将"该用哪个大模型翻译？"从猜测转变为基于度量的自动决策——且随着底层模型格局变化，工具性能自然提升，无需修改代码。

此外，系统刻意限定了范围。项目早期曾支持本地运行时供应商（如 Ollama、LlamaCpp）及本地 SSH 翻译路径；后续桥接阶段，我 **移除了** 这些选项，并彻底移除本地 SSH 翻译功能。如今选择这些路径将直接返回"不再支持"的错误提示，而非运行降级的本地模型。这同样是诚实原则在路线图中的体现：我宁可砍掉一条路径，也不愿让其悄然输出低质量结果。

## 最难部分的解决之道

架构的基石是 **桥接包**（`pkg/bridge`），它连接翻译模块与 LLMsVerifier。翻译器不再硬编码供应商偏好，而是向桥接模块请求当前最强的已验证模型及按评分排序的回退链路，然后基于此进行翻译。这正是"无需选择供应商"在实践中成立的关键，也是"禁止静默回退"规则的执行点——桥接模块会返回明确的错误，而非替换为本地模型。

为实现实时监控，我将功能拆分至专用模块：`websocket` 中心负责连接与广播管理，`events` 系统定义并发送事件流，`sshworker` 包处理远程执行。独立的监控服务器消费 WebSocket 事件流并驱动面板，使翻译运行与监控解耦——CLI 发送事件，服务器分发，任意数量的面板客户端均可订阅并自动重连。

在分布式处理方面，SSH 工作节点层负责安全连接与远程执行，同时通过同一事件系统回传进度，因此远程翻译在面板视角下与本地翻译无异。整个系统通过 REST API 暴露，支持 HTTP/3，并由全面的 WebSocket 监控测试套件支撑——因为一个无法观测的翻译系统，就是一个无法信任的翻译系统。

## 후크

자체 메일 서버를 운영하는 것은 시스템 관리 분야에서 가장 고통스러운 작업 중 하나로 악명이 높습니다. SMTP, IMAP, POP3, TLS, DKIM, SPF, 방화벽, 인증서, SELinux, 그리고 서로 협력해야만 제대로 작동하는 수십 개의 데몬들까지. 메일 서버 팩토리(Mail Server Factory)의 홍보 문구는 단도직입적입니다. **"보스처럼 메일 서버를 운영하세요."** 원하는 설정을 간단한 JSON 파일로 작성하면 팩토리가 이를 해석해 목표 머신에 완전히 경화된 도커 기반 스택을 구축해줍니다.

## 왜 매력적인가

매력적인 아이디어는 메일 서버 배포를 *컴파일* 문제로 접근한다는 점입니다. JSON 설정은 소스 코드이고, 팩토리는 컴파일러이며, 그 결과물은 목표 OS에서 실행되는 느슨하게 결합된 도커 기반 메일 스택입니다. 각 구성 요소가 느슨하게 결합되고 컨테이너화되어 있기 때문에, 이는 나중에 확장하기 위한 깔끔한 기반을 제공하며, 아무도 건드리지 못하는 수동으로 편집된 설정 파일들의 얽힌 실타래와는 다릅니다.

이 프로젝트를 "재미있다"에서 "진지하다"로 격상시키는 것은 바로 그 범위입니다. 팩토리는 12가지 서로 다른 연결 방식을 통해 스택을 제공할 수 있습니다. SSH, 도커, 쿠버네티스, AWS SSM, 애저 시리얼 콘솔, GCP OS 로그인, 리브버트, 커스텀 프로토콜, 데이터베이스, 파일 시스템, 클라우드 프로바이더, 컨테이너 런타임까지. 그리고 25개의 리눅스 배포판을 지원하는데, 대부분의 도구가 무시하는 배포판들도 포함됩니다. 서구권의 주요 배포판(우분투, 데비안, 센트OS, 페도라, 알마리눅스, 로키, 오픈수세)은 물론 러시아 배포판(ALT, 아스트라, 로사), 중국 배포판(오픈오일러, 오픈카일린, 딥인)까지 아우릅니다. 이러한 폭넓은 지원은 명백한 메시지를 담고 있습니다. **당신의 메일 서버, 당신의 하드웨어, 당신의 관할권.**

## 어려운 문제들

첫 번째 어려운 문제는 "동일한 설치"가 결코 동일하지 않다는 점입니다. 패키지 이름, init 시스템, 방화벽 프론트엔드, SELinux 동작 방식은 25개 배포판마다 제각각입니다. 팩토리는 실제 보안 프레임워크를 탑재하고 있습니다. `CertificateValidator`, `DockerCredentialsManager`, `SELinuxChecker`, `PasswordValidator`, 그리고 `ConnectionPool`이 바로 그것입니다. 이러한 요소들은 플랫폼마다 다르게 동작하기 때문에 그 어떤 것도 가정할 수 없기 때문입니다.

두 번째 문제는 연결 계층 자체입니다. SSH를 통한 접근은 AWS SSM, 애저 시리얼 콘솔, 리브버트를 통한 접근과는 전혀 다릅니다. 각 전송 방식은 고유한 인증 방식, 지연 시간, 실패 모드를 가지고 있습니다. 12가지 연결 방식을 하나의 설치 파이프라인으로 추상화하여 설치 단계가 바이트가 어떻게 전달되는지 신경 쓰지 않도록 만드는 것이 이 프로젝트의 핵심 아키텍처입니다.

세 번째 문제는 애초에 깨끗한 머신을 확보하는 것입니다. 팩토리는 preseed, kickstart, cloud-init, autoyast를 통해 모든 지원 배포판의 무인 OS 설치를 자동화하고, `scripts/qemu_manager.sh`를 통해 QEMU 가상 머신을 제어하며, ISO 파일을 체크섬 검증과 함께 관리합니다. 여기에 엔터프라이즈급 기능까지 더해졌습니다. 네트워크 공유에서 누락된 이미지를 가져오고, 반대로 누락된 이미지를 업로드하며, 최후의 수단으로 인터넷 다운로드에 의존하는 양방향 SMB 캐시입니다.

## 게임 체인저인 이유

이 프로젝트의 진정한 혁신은 자가 호스팅 메일을 *재현 가능하고 감사 가능하게* 만든다는 점입니다. 이메일 인프라는 데이터로 기술되고 동일하게 재구축되어야 하는 대상이지, 새벽 2시에 기억에 의존해 조립하는 대상이 아닙니다. 스택이 JSON으로 정의되고 보안 태세가 코드로 강제되면—AES-256-GCM 암호화, 최소 12자 이상의 SSH 키 암호 구문 필수, 25/587/465/993/995 포트에 대한 자동 방화벽 규칙, 인증서 검증 및 HSTS가 적용된 TLS, 감사 로깅, RBAC—"프로덕션급 메일 서버"는 수 주에 걸친 프로젝트가 아니라 검토, 버전 관리, 재실행이 가능한 설정으로 변모합니다.

또한 이 프로젝트는 이상적인 목표가 아니라 엄격한 검증을 거쳤습니다. 439개 테스트를 100% 통과했으며, SonarQube 품질 게이트도 100% 충족했습니다. 실제 SMTP/IMAP/POP3 동작을 배포판별로 테스트하는 프레임워크는 물론, 처리량과 지연 시간 측정까지 포함됩니다. 사용자의 이메일을 맡길 도구로서 신뢰성이야말로 가장 중요한 기능입니다.

## 가장 어려운 부분을 해결한 방법

저는 *무엇을 구축할지*, *어떻게 목표에 도달할지*, *어떤 OS를 사용할지*를 분리했습니다. JSON 설정은 의도를 기술하며, 전송 추상화 계층은 12가지 연결 방식을 하나의 인터페이스로 숨겨 설치 단계가 SSH든 AWS SSM이든 동일한 논리적 작업을 수행하도록 합니다. 또한 배포판별 로직은 격리되어 있어 25개 배포판의 매트릭스가 한 곳에 집중되어 있으며, 각 단계로 새어나가지 않습니다. 이러한 계층화 덕분에 새로운 배포판이나 연결 방식을 추가해도 코드 전체에 영향을 미치지 않습니다.

보안 측면에서는 운영자의 선의에 맡기지 않았습니다. 프레임워크가 강제하는 부분—SELinux *확인*, 인증서 *검증*, 도커 자격 증명 *관리*, 취약한 비밀번호와 암호 없는 SSH 키 *거부*—은 올바른 배포가 기본값이 되도록 합니다. 셸은 무거운 프로비저닝에 적합한 도구로 활용하고, 코틀린/JVM은 오케스트레이션과 보안 프레임워크를 담당하며, 도커는 모든 구성 요소를 느슨하게 결합합니다.

또한 대부분의 설치 도구가 회피하는 "깨끗한 머신" 부트스트래핑 문제도 해결했습니다. 지원하는 모든 배포판의 무인 설치 자동화, 테스트 대상 가상 머신을 구동하는 QEMU 자동화, 그리고 동일한 이미지를 25번 다시 다운로드하지 않도록 하는 양방향 SMB ISO 캐시까지. 모든 ISO는 출처와 상관없이 체크섬 검증을 거칩니다. 그 결과, 베어 메탈(또는 빈 VM)에서 시작해 하나의 파일로 경화되고 테스트된 실행 중인 메일 서버까지 단번에 구축할 수 있는 도구가 탄생했습니다.
